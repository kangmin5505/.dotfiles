현재 브랜치의 커밋 내역을 분석하여 GitHub PR 본문을 **한국어**로 작성해줘.

**베이스 브랜치 결정 (최우선 수행):**
1. 사용자가 인자로 베이스 브랜치를 전달한 경우 (`$ARGUMENTS`), 해당 브랜치를 베이스로 사용해줘.
2. 인자가 없으면, `git log --oneline -1 --merges` 등으로 가장 가까운 부모 브랜치를 추론하거나, 사용자에게 베이스 브랜치를 질문해줘.

**커밋 비교 방법:**
- `git merge-base HEAD <베이스 브랜치>` 로 분기점을 찾고,
- `git log --oneline --no-merges <분기점>..HEAD` 로 이 브랜치의 고유 커밋만 추출해줘.
- `git diff <분기점>..HEAD` 로 실제 코드 변경분을 분석해줘.

**분석 및 작성 원칙:**
1. **고유 변경분만 분석**: 위에서 추출한 커밋과 diff만 대상으로 해줘. 베이스 브랜치에 이미 있는 내용은 절대 포함하지 마.
2. **기술적 의도 파악**: 단순히 "무엇을 수정함"이 아니라, 변경된 코드의 맥락을 통해 이 작업이 지향하는 **기술적 핵심 설계와 의도**를 스스로 도출해줘.
3. **간결하고 명료하게**: 서술식 문장을 지양하고, 핵심만 추린 짧은 불렛 포인트로 작성해줘. 한 불렛이 2줄을 넘기지 마.
4. **파일 출력**: 작성 완료된 PR 본문을 `/tmp/pr-desc.md` 파일에 저장해줘. 채팅에는 파일 경로만 안내하고, 서론/결론은 생략해줘.
5. **내부 참조 우선**: 동작을 설명할 때 외부 시스템에 비유하지 말고, CUBRID 내부에 동일/유사한 기능이 있으면 그것을 참조할 것.
6. **핵심 설계 변경만 기재**: Implementation에서 사소한 방어 코드(에러 전파 보강, null 체크 추가, assert 추가 등)는 별도 불렛으로 올리지 말 것. 아키텍처나 데이터 흐름에 영향을 주는 변경만 기술할 것.
7. **Remarks는 비직관적 선택만**: 리뷰어가 "왜 이렇게 했지?"라고 의문을 가질 만한 비직관적 설계 선택만 기재할 것. 당연한 사실(추정값이므로 근사값 등)이나 코드를 보면 바로 알 수 있는 내용은 제외할 것.

**형식:**
[Jira Ticket URL]
### Purpose
- (이 작업의 핵심 목적을 1~2개 불렛으로)

### Implementation
- (주요 변경 사항을 기술적 관점에서 불렛 포인트로 나열)
- (세부 항목이 필요하면 들여쓰기 불렛 사용)

### Remarks
- (리뷰어가 알아야 할 주의사항, 제약, 사이드이펙트만 기재)
- (베이스 브랜치, 변경 파일 수 등 git에서 바로 확인 가능한 정보는 제외)
